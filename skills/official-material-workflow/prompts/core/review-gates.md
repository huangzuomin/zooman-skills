### ReviewReport 固定格式

```yaml
review_result: pass / warning / fail
risk_level: S0 / S1 / S2
summary:
issues:
  - check_id:
    severity:
    issue:
    evidence:
    suggestion:
human_confirm_required:
```

### 严重级

- S0：阻断成稿或发布，必须退回补充材料、重路由或人工处理。
- S1：必须人工确认，可形成草稿但不得标记为 final。
- S2：可自动提醒或修正，修正后仍建议人工抽检。

### S0 阻断项

- 文种明显错误。
- 行文方向错误。
- 报告、简报、汇报材料中夹带“请批准”“请批复”等请示事项。
- 请示包含多个独立审批事项。
- 核心结论无 E4 以上证据支撑，或仅由 E0/E1/E2 支撑。
- 涉及其他部门职责但未提供会签、征求意见或共同决定依据，却写成联合决定。
- 涉密信息拟公开，或公开属性与密级冲突。
- 涉规范性文件、政策措施、权利义务影响事项，但无法律政策依据。

### S1 人工确认项

- 数据缺来源、缺时间、缺统计口径。
- 公开属性缺失。
- 保密等级、内部流转范围或发布路径不明确。
- 涉及多部门事项但协商状态不明确。
- 对比型判断缺同比、环比、横比或其他基线。
- 重要人名、地名、机构简称、文件标题或日期需核对。

### S2 提醒项

- 标点、数字、计量单位、层级编号不够规范。
- 句子过长、套话过多或讲话稿不便口头朗读。
- 附件、附图、附注、版记等结构栏位与模板轻微不一致。

### 输出规则

- 存在任一 S0 时，`review_result` 必须为 `fail`，不得输出“正式定稿”口径。
- 存在 S1 且无 S0 时，`review_result` 为 `warning`，必须列出 `human_confirm_required`。
- 审稿报告必须区分事实问题、文种问题、程序问题、格式问题和语言问题。
